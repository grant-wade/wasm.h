# wasi.h Implementation Plan

Single-header C99 library implementing the WebAssembly Component Model, Canonical ABI, and WASI 0.2+ interfaces. Depends on `wasm.h` for core module execution.

**Baseline:** `wasm.h` passes the full spectest suite and implements Wasm 3.0 core spec features including GC, exception handling, SIMD, Memory64, multiple memories, tail calls, and typed function references.

---

## Architecture

```
┌────────────────────────────────────────────┐
│                 wasi.h                     │
│                                            │
│ ┌──────────┐  ┌───────────┐  ┌───────────┐ │
│ │Component │  │ Canonical │  │   WASI    │ │
│ │  Binary  │  │    ABI    │  │  Worlds   │ │
│ │  Parser  │  │ Lift/Lower│  │ (0.2/0.3) │ │
│ └────┬─────┘  └─────┬─────┘  └─────┬─────┘ │
│      │              │              │       │
│ ┌────┴──────────────┴──────────────┴─────┐ │
│ │         Component Runtime              │ │
│ │   (instantiation, linking, handles)    │ │
│ └────────────────┬───────────────────────┘ │
└──────────────────┼─────────────────────────┘
                   │ depends on
┌──────────────────┼─────────────────────────┐
│              wasm.h                        │
│  (core module parse, validate, interpret)  │
└────────────────────────────────────────────┘
```

**Public API pattern:**

```c
#define WASI_IMPL
#include "wasi.h"  // pulls in wasm.h

wasi_runtime_t wrt;
wasi_init(&wrt, NULL);                    // internally calls wasm_init
wasi_bind_cli(&wrt, argc, argv, envp);
wasi_bind_filesystem(&wrt, &fs_cfg);

wasi_component_t *comp = wasi_load_component(&wrt, bytes, len);
wasi_instance_t  *inst = wasi_instantiate(comp);
wasi_call(inst, "wasi:cli/run@0.2.0", "run", ...);

wasi_free_instance(inst);
wasi_free_component(comp);
wasi_destroy(&wrt);
```

## Milestone 1: Component Binary Parser

**Goal:** Parse the component binary format, extract embedded core modules, and recover the component's import/export structure.

### Background

A component binary uses the same `\0asm` magic but a different layer byte (0x0d for component vs 0x01 for core module). It contains:

- Core module sections (embedded complete core `.wasm` modules)
- Component type sections (WIT-derived type definitions)
- Import/export sections (interface-level imports and exports)
- Canon sections (canonical function definitions — lift/lower/resource operations)
- Instance sections (instantiation instructions wiring modules together)
- Alias sections (aliasing exports from child instances)
- Start section (component-level start function)

### Deliverables

- Binary parser for the component section format
- Extraction of embedded core modules into `wasm_module_t*` via `wasm_load`
- Parse component type index space: defined types, function types, component types, instance types
- Parse component imports and exports with full interface names (e.g., `wasi:filesystem/types@0.2.0`)
- Parse canonical function definitions (`canon lift`, `canon lower`, `canon resource.new`, etc.)
- Parse instantiation, alias, and start sections
- Diagnostic output: `wasi_component_info(comp)` dumps structure summary (similar to `wasm info`)

### Validation

- Round-trip test: load component binaries produced by `wasm-tools component new`, verify extracted core modules pass existing validation
- Parse components generated from Rust `wasm32-wasip2` target
- Parse components generated by `wasm-tools compose`
- Compare parsed structure against `wasm-tools component wit` output

### Test corpus

- Minimal hand-crafted components via `wasm-tools component new`
- Rust `wasm32-wasip2` hello-world component
- Multi-module composed components via `wasm-tools compose`

### Estimated effort: 3–4 weeks

## Milestone 2: Canonical ABI — Scalar Types & Strings

**Goal:** Implement canonical ABI lifting and lowering for scalar types and strings, sufficient to call component exports that take/return integers, floats, bools, chars, and strings.

### Background

The canonical ABI defines how WIT-level types map to core Wasm values and linear memory:

- Scalars (`bool`, `u8`–`u64`, `s8`–`s64`, `f32`, `f64`, `char`): direct mapping to core i32/i64/f32/f64 with validation (e.g., `bool` must be 0 or 1, `char` must be valid Unicode scalar)
- `string`: lowered as `(i32 pointer, i32 length)` in linear memory, UTF-8 encoded
- Strings require calling the component's `cabi_realloc` export for allocation on the callee side

### Deliverables

- `canon lift` implementation for scalar types: validate and convert core Wasm values to WIT values
- `canon lower` implementation for scalar types: convert WIT values to core Wasm values
- String lifting: read `(ptr, len)` from core Wasm, copy UTF-8 bytes out of linear memory
- String lowering: allocate via `cabi_realloc`, write UTF-8 bytes into linear memory, pass `(ptr, len)`
- `cabi_realloc` export discovery and invocation
- `wasi_call` API that accepts C-side values, lowers them, calls the core function, and lifts results
- Post-return cleanup dispatch (`cabi_post_*` exports)

### Validation

- Test component exporting `func(s: string) -> string` — round-trip a string through component boundary
- Test all scalar types: boundary values (i32 min/max, NaN, Unicode edge cases)
- Verify `cabi_realloc` is called with correct alignment and size arguments
- Memory leak check: post-return functions are called

### Estimated effort: 3–4 weeks

## Milestone 3: Canonical ABI — Compound Types

**Goal:** Complete the canonical ABI for all WIT types: lists, records, tuples, variants, enums, options, results, and flags.

### Deliverables

- **Lists:** `list<T>` lowered as `(i32 pointer, i32 length)` with element layout per inner type. Nested lists require recursive realloc.
- **Records/Tuples:** Flattened into sequential core Wasm params/results up to the `MAX_FLAT_PARAMS` (16) / `MAX_FLAT_RESULTS` (1) thresholds; spilled to linear memory via a return-area pointer when exceeding limits.
- **Variants/Enums:** Discriminant as `i32`, payload follows per-case layout rules with alignment padding.
- **Option\<T\>:** Sugar for `variant { none, some(T) }`.
- **Result\<T, E\>:** Sugar for `variant { ok(T), err(E) }`.
- **Flags:** Bit-packed into one or more `i32` values.
- Flattening/spilling logic: determine when args/results exceed flat limits and switch to memory-based passing via `retptr`.
- Alignment and padding calculations per the canonical ABI spec.

### Validation

- Test component with `func(r: record { a: u32, b: string, c: list<u8> }) -> result<string, string>`
- Verify spilling: function with >16 params hits the memory-based path
- Verify single-result spilling: function returning a record with >1 flat field uses retptr
- Fuzz: generate random WIT types, compile Rust components, round-trip values

### Estimated effort: 4–5 weeks

## Milestone 4: Resource Types & Handle Tables

**Goal:** Implement resource lifecycle management — handle tables, ownership, borrowing, and destructor dispatch.

### Background

WIT `resource` types map to `i32` handle indices in core Wasm. The component runtime maintains per-component handle tables that track:

- **Own handles:** caller transfers ownership; handle is consumed on use
- **Borrow handles:** caller lends access; handle remains valid for the call duration
- **Destructors:** `canon resource.drop` invokes the resource's destructor when the last own handle is dropped

### Deliverables

- Per-component handle table: `wasi_handle_table_t` mapping `i32` indices to host-side resource pointers
- `canon resource.new` implementation: allocate a handle, associate with host resource
- `canon resource.rep` implementation: retrieve the representation from a handle
- `canon resource.drop` implementation: invoke destructor, remove from table
- Own vs borrow tracking: borrow handles valid only for call duration, trap on use-after-return
- Handle transfer between components during cross-component calls
- Host-side resource type registration API:

```c
wasi_resource_type_t rt = wasi_define_resource(&wrt, "file-descriptor", my_dtor, userdata);
```

### Validation

- Create a resource on host side, pass own handle to component, verify destructor called on drop
- Borrow handle: verify trap if component tries to drop a borrowed handle
- Cross-component handle transfer: component A creates resource, passes own to component B
- Handle table exhaustion: verify clean error on overflow

### Estimated effort: 3–4 weeks

## Milestone 5: Component Instantiation & Linking

**Goal:** Full component instantiation — resolve imports, link core modules, wire up canonical functions, and execute the component start section.

### Background

A component's instantiation section contains procedural instructions:

1. Instantiate core modules with resolved imports
2. Create instances from other components (nested composition)
3. Alias exports from instances into the current scope
4. Define canonical lift/lower wrapper functions
5. Export interfaces from the top-level component

### Deliverables

- Import resolution: match component imports against host-provided interfaces and other component exports by interface name + version
- Core module instantiation: invoke `wasm_load` for each embedded core module, providing resolved imports
- Alias resolution: alias core exports, component exports, and outer aliases
- Canonical function wiring: for each `canon lift` and `canon lower`, create thunks that perform ABI translation and dispatch to the underlying core function
- Component start execution
- Multi-module support: components containing multiple core modules wired together
- Error reporting: clear diagnostics for unresolved imports, type mismatches, instantiation failures

### Validation

- Single-module component: Rust `wasm32-wasip2` hello-world instantiates and runs
- Multi-module component: `wasm-tools compose` output with two modules linked together
- Missing import detection: clean error message naming the unresolved interface
- Nested component instantiation

### Estimated effort: 4–5 weeks

## Milestone 6: wasi:cli World — Hello World

**Goal:** Implement enough WASI 0.2 interfaces to run a `wasi:cli/command` component that prints to stdout.

### Interfaces required

- `wasi:cli/environment@0.2.0` — `get-environment`, `get-arguments`, `initial-cwd`
- `wasi:cli/stdin@0.2.0`, `wasi:cli/stdout@0.2.0`, `wasi:cli/stderr@0.2.0` — stream resources
- `wasi:cli/exit@0.2.0` — `exit(result)`
- `wasi:io/streams@0.2.0` — `output-stream` resource with `blocking-write-and-flush`, `write`, `subscribe`
- `wasi:io/poll@0.2.0` — `pollable` resource and `poll` function (minimal: single-pollable blocking)
- `wasi:io/error@0.2.0` — error resource type
- `wasi:clocks/wall-clock@0.2.0` — `now`, `resolution` (needed by Rust stdlib init)
- `wasi:clocks/monotonic-clock@0.2.0` — `now`, `resolution`, `subscribe-duration`

### Deliverables

- Host interface implementations for all of the above
- Configuration API:

```c
wasi_cli_config_t cli = {
    .argc = argc,
    .argv = argv,
    .envp = envp,
    .stdin_fd  = STDIN_FILENO,
    .stdout_fd = STDOUT_FILENO,
    .stderr_fd = STDERR_FILENO,
};
wasi_bind_cli(&wrt, &cli);
```

- Stream resource implementation backed by file descriptors
- Minimal pollable implementation (blocking only — no async event loop yet)

### Validation

- Rust `wasm32-wasip2`: `fn main() { println!("Hello, world!"); }` runs and produces output
- Rust component reading env vars and args
- Exit codes propagated correctly
- Clock queries return reasonable values

### 🎉 This is the "Hello World from a component" milestone

### Estimated effort: 3–4 weeks

## Milestone 7: wasi:filesystem

**Goal:** Implement filesystem interfaces sufficient for real programs to read/write files and traverse directories.

### Interfaces

- `wasi:filesystem/types@0.2.0` — `descriptor` resource, `stat`, `open-at`, `read-via-stream`, `write-via-stream`, `readdir`, `create-directory-at`, `remove-directory-at`, `unlink-file-at`, `rename-at`, `metadata-hash-at`, `sync`, etc.
- `wasi:filesystem/preopens@0.2.0` — `get-directories` returning pre-opened directory handles

### Deliverables

- `descriptor` resource type backed by host file descriptors
- Capability-based access: preopens define the sandbox boundary, all path resolution relative to preopens
- Path resolution with `..` traversal prevention (no sandbox escape)
- Stream integration: `read-via-stream` / `write-via-stream` return `input-stream` / `output-stream` resources
- Preopen configuration API:

```c
wasi_fs_config_t fs = {
    .preopens = (wasi_preopen_t[]){
        { .guest_path = "/", .host_path = "./sandbox" },
        { .guest_path = "/tmp", .host_path = "/tmp/wasi-sandbox" },
    },
    .num_preopens = 2,
    .readonly = false,
};
wasi_bind_filesystem(&wrt, &fs);
```

- `stat`, `readdir`, `open-at` with `o-flags` (create, exclusive, truncate, directory)
- `metadata-hash-at` for content-addressable use cases

### Validation

- Rust component that creates a file, writes data, reads it back, verifies contents
- Directory listing matches host directory contents
- Sandbox escape attempt (e.g., `../../etc/passwd`) is blocked
- Symlink following respects sandbox boundary
- Read-only mode enforced when configured

### Estimated effort: 4–5 weeks

### Deliverables

## Milestone 10: WASI 0.3 Async Primitives (Stretch)

**Goal:** Support WASI 0.3's native async model — `future<T>`, `stream<T>`, and async function signatures.

### Background

WASI 0.3 adds async at the canonical ABI level. Components can export and import async functions. The runtime is responsible for managing a task queue and dispatching between suspended component calls.

### Deliverables

- `future<T>` type: canonical ABI lift/lower for futures
- `stream<T>` type: canonical ABI for typed byte/value streams with backpressure
- Async canonical ABI: `canon lift` with `async` flag, `canon lower` with `callback` option
- Task management: per-instance task table, suspend/resume dispatch
- `subtask.drop`, `stream.new`, `stream.read`, `stream.write`, `future.new`, `future.read`, `future.write` built-in functions
- Host-side async integration point:

```c
// Host provides an event loop callback
wasi_set_async_dispatch(&wrt, my_event_loop, userdata);
```

- Refactor `wasi:io/streams` and `wasi:http` to use 0.3 async interfaces

### Validation

- Async HTTP handler: component handles concurrent requests without blocking
- Stream backpressure: slow consumer pauses fast producer
- `wasm-tools` 0.3-format components from Rust `wasm32-wasip3` (when available)
- Wasmtime 37+ parity testing for async component behavior

### Estimated effort: 6–8 weeks

## Testing Strategy

### Test corpus sources

- **wasm-tools:** Generate components from WIT definitions, compose multi-module components, validate parser output against `wasm-tools component wit` and `wasm-tools dump`
- **Rust wasm32-wasip2:** Real compiler output from trivial to complex programs
- **TinyGo wasip2:** Secondary language validation
- **Spectest-style harness:** Extend existing spectest runner to handle component binaries, with expected-output assertions

### Continuous validation

- Every milestone adds tests to CI
- Components from previous milestones continue to pass as new features are added (regression)
- Valgrind/ASan runs on every PR
- Comparison testing against Wasmtime output where feasible (same component, same inputs, same results)

### Fuzz targets

- Component binary parser: feed random bytes, verify no crashes
- Canonical ABI round-trip: generate random WIT types, compile Rust components, round-trip values through lift/lower
- Handle table: random sequence of new/drop/borrow operations

## Open Design Questions

1. **Async model (M10):** The host-side async integration could be a simple poll loop, integration with `epoll`/`kqueue`, or a callback-based model. Decision deferred until M9 when the sync implementation is stable.

2. **HTTP implementation:** M8's outgoing HTTP handler needs a client implementation. Options: minimal built-in HTTP/1.1, delegate to host callback, or optional dependency on a library like `libcurl`. Host callback is likely the cleanest for an embeddable library.

3. **WASI version negotiation:** Components may target different WASI versions (0.2.0, 0.2.1, 0.3.0). The runtime needs a strategy for handling minor version differences — likely "implement the superset and accept any compatible version."

4. **wasm2api integration:** `wasm2api` currently generates wrappers for core module exports. A natural extension would be generating typed C wrappers for component exports, using `wasi.h` under the hood. This could be a follow-on project after M6.
