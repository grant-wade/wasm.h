Here is a comprehensive, phased project plan for building `wasm2api.c`. 

---

### [DONE] Phase 1: The CLI Foundation & Type Mapper
**Goal:** Create a standalone CLI tool that reads a `.wasm` file, parses its signature using `wasm.h`, and correctly maps WebAssembly types to native C types.

1. **CLI Boilerplate:**
   * Accept an input file (e.g., `math.wasm`).
   * Accept an output prefix (e.g., `math`).
   * Use `wasm_init()` and `wasm_load()` to parse the binary.
2. **The Type Mapper Function:**
   * Write a helper function inside `wasm2api.c` that translates `wasm_valtype_t` to C string representations.
   * `WASM_TYPE_I32` -> `"int32_t"`
   * `WASM_TYPE_I64` -> `"int64_t"`
   * `WASM_TYPE_F32` -> `"float"`
   * `WASM_TYPE_F64` -> `"double"`
   * `WASM_TYPE_EXTERNREF` -> `"void*"`
3. **Multi-Value Return Handling:**
   * C doesn't support multiple return values. If a Wasm function returns 0 items, the C return type is `void`. If it returns 1 item, return it directly. If it returns >1 item, the C return type is `void` and the generated arguments must include *pointers* for the outputs (e.g., `int32_t* out_1, float* out_2`).

---

### [DONE] Phase 2: Generating the Header (`.h`)
**Goal:** Generate a clean, typed interface that host developers will `#include`.

Iterate over the Wasm module exports using `wasm_export_count()`. For every `WASM_EXPORT_FUNC`, generate the C prototype.

**Target Output Example (`math.h`):**
```c
/* AUTO-GENERATED BY wasm2api */
#pragma once
#include <stdint.h>
#include "wasm.h"

// Opaque context holding the loaded module and cached indices
typedef struct math_ctx_t math_ctx_t;

// Lifecycle
math_ctx_t* math_init(wasm_runtime_t* rt, const uint8_t* wasm_bytes, size_t len);
void math_free(math_ctx_t* ctx);

// Exported Functions
int32_t math_add(math_ctx_t* ctx, int32_t a, int32_t b);
void math_div_mod(math_ctx_t* ctx, int32_t a, int32_t b, int32_t* out_div, int32_t* out_mod);
```

---

### [DONE] Phase 3: Generating the Implementation (`.c`)
**Goal:** Generate the C source file that does the heavy lifting of packing/unpacking `wasm_value_t` arrays and utilizing `wasm_call_index` for maximum speed.

1. **The Context Struct:**
   * Generate a struct containing the `wasm_module_t*` and `uint32_t` fields to cache the index of every exported function. (This avoids slow string lookups at runtime).
2. **The `_init` Function:**
   * Generate code that calls `wasm_load()`.
   * Generate sequential `wasm_find_export()` calls to populate the cached indices.
3. **The Function Wrappers:**
   * For each function, generate the packing logic.

**Target Output Example (`math.c`):**
```c
/* AUTO-GENERATED BY wasm2api */
#include "math.h"
#include <stdlib.h>

struct math_ctx_t {
    wasm_module_t* mod;
    uint32_t idx_add;
    uint32_t idx_div_mod;
};

math_ctx_t* math_init(wasm_runtime_t* rt, const uint8_t* bytes, size_t len) {
   math_ctx_t* ctx = malloc(sizeof(math_ctx_t));
    ctx->mod = wasm_load(rt, bytes, len);
    if (!ctx->mod) { free(ctx); return NULL; }
    
    // Cache indices! O(1) calls later.
    wasm_find_export(ctx->mod, "add", NULL, &ctx->idx_add);
    wasm_find_export(ctx->mod, "div_mod", NULL, &ctx->idx_div_mod);
    return ctx;
}

int32_t math_add(math_ctx_t* ctx, int32_t a, int32_t b) {
    wasm_value_t args[2] = { wasm_i32(a), wasm_i32(b) };
    wasm_value_t results[1];
    
    // Fast path: call by index
    wasm_call_index(ctx->mod, ctx->idx_add, args, 2, results, 1);
    return results[0].of.i32;
}

// Example of Multi-return handling
void math_div_mod(math_ctx_t* ctx, int32_t a, int32_t b, int32_t* out_div, int32_t* out_mod) {
    wasm_value_t args[2] = { wasm_i32(a), wasm_i32(b) };
    wasm_value_t results[2];
    
    wasm_call_index(ctx->mod, ctx->idx_div_mod, args, 2, results, 2);
    *out_div = results[0].of.i32;
    *out_mod = results[1].of.i32;
}
```

### Phase 4: Import Binding Generation (The "Reverse" FFI)
**Goal:** If the WebAssembly module *requires* imports from the host (like `env.print_debug` or `env.get_time`), the generator should create a struct of function pointers for the user to fill out, and auto-generate the `wasm_host_func_t` wrappers.

1. **Introspect Imports:** In `wasm.h`, you'll need to expose a small helper: `wasm_import_count(mod)` and `wasm_import_info(mod, idx)`. (Currently, you track these, but might need to expose them publicly).
2. **Generate an Interface Struct in the `.h`:**
   ```c
   typedef struct {
       void (*print_debug)(int32_t val, void* userdata);
       float (*get_time)(void* userdata);
       void* userdata;
   } math_imports_t;
   
   // Updated Init function requires the imports!
   math_ctx_t* math_init(..., const math_imports_t* imports);
   ```
3. **Generate the Dispatchers in the `.c`:**
   Generate the `wasm_host_func_t` callbacks that unpack `wasm_value_t`, call the user's C function pointer, and pack the result back.

---

### Phase 5: Memory Access & "String" Ergonomics (Advanced)
**Goal:** Provide safe, named wrappers for exported memory and globals.

1. **Globals:** If the module exports a global named `MAX_SPEED`, generate `float math_get_MAX_SPEED(math_ctx_t* ctx);`.
2. **Memory Pointers:** If the module exports memory, generate a helper to get a direct pointer:
   ```c
   uint8_t* math_memory(math_ctx_t* ctx);
   ```
3. **The "String/Pointer" Heuristic (Optional but cool):** 
   If you have a function `print_string(i32 ptr, i32 len)`, Wasm doesn't know it's a string. However, if the user passes a CLI flag to `wasm2api` like `--string-args print_string:0`, the generator can alter the C signature to `void print_string(const char* str)` and automatically generate the code to copy the string into the Wasm memory before calling.

---

### Execution Plan

To build this, you don't need to change `wasm.h` much, except maybe adding a couple of introspection getters if they are missing (like inspecting *Imports* before a module is fully linked/loaded).

1. **Create `wasm2api.c`** in your repository.
2. Implement **Phases 1 & 2**. Run it on a simple Wasm file and verify the `.h` file prints correctly to `stdout`.
3. Implement **Phase 3**. Redirect output to write to actual `.c` and `.h` files. Compile a test program linking against the generated `.c` file.
4. (Optional) Implement **Phase 4** for modules that require Host imports.
